Step 1: Understand Your Home Network Like a Responsible Adult‚Äã

Most homes today have more than one way to get online, even if you don‚Äôt realize it. Computer networks are everywhere‚Äîbroadband, mobile data, and hotspots.

Count the Networks‚Äã

Total Networks at home = 1 broadband + each phone with an active mobile data plan

That means your child‚Äôs screen time could sneak in through any of these entry points. First step: identify and isolate them.

Basic Network Topology (Before Changes)‚Äã

Internet ‚Üí ISP Router (Wi-Fi & LAN) ‚Üí All Devices ‚Üí your child

This gives the ISP‚Äôs router full control over DNS, firewall rules, and device access. Not ideal for a security-conscious parent.

Smart phones + data packs ‚Üí your child

Data packs by carrier don‚Äôt give any parental control tools.

We need to target the network-level filtering, which is generally good enough to block ads, parked domains, and time wasters.

Step 2: Fix the Manhole ‚Äî Lock Down Your Broadband‚Äã

Why ISP Routers Are a Problem‚Äã

  • Most ISP routers are locked down‚ÄîDNS settings greyed out
  • Firmware is often outdated and insecure
  • Limited or no parental control features
  • No custom firewall or VLAN options

  1. Disabled Wi-Fi on the ISP router
    Let it act as a basic modem or Layer 2 switch.
  2. Introduced My Own Router (Netgear)
    Plugged into the ISP box via Ethernet (LAN to WAN). This separates my internal LAN from the ISP‚Äôs network.
  3. Enabled NAT, DHCP, and DNS on My Router
    Configured to issue private IPs (e.g., 192.168.1.x) and point all DNS queries to OpenDNS servers:
    • 208.67.222.222
    • 208.67.220.220
  4. Registered with OpenDNS
    Created a free account, added my dynamic IP, and enabled category-level filtering. You can block:
    • Social networks
    • Video sharing
    • Adult content
    • Gaming, shopping, forums‚Äîup to you
  5. Tested the Setup
    Verified DNS override by visiting:
    https://welcome.opendns.com

Updated Network Topology‚Äã

All devices now talk to the internet via my firewall + DNS filters, not the ISP‚Äôs.

Step 3: Taming the Phones‚ÄîEspecially the Ones With Data Packs‚Äã

Broadband is fixed, but what about mobile data on phones? These are the real backdoors in your computer networks.

Simple But Effective Plan‚Äã

  • Recharge only when needed (no monthly auto-renewal of data-heavy packs)
  • Use mobile data only for OTPs, emergency maps, and short bursts

The Engineer‚Äôs Setup‚Äã

I needed something robust and centrally managed.

  1. For Broadband Devices ‚Äì I use OpenDNS via the router.
  2. For Mobile Data (4G/5G) Devices ‚Äì I use NextDNS via Android/iPhone‚Äôs Private DNS feature.

  • Supports DNS-over-TLS (DoT) and DNS-over-HTTPS (DoH)

  • Works even over mobile data, friends‚Äô hotspot
  • One profile can be shared across multiple devices
  • For entire family, basic plan is worth it. Free plan is limited.

  • Go to: Settings > Network & Internet > Private DNS
  • Choose: Private DNS provider hostname
  • Enter: your-profile-id.dns.nextdns.io (from nextdns.io dashboard)
  • Save and test at: https://test.nextdns.io

Now even on 4G, your DNS traffic is encrypted and filtered.

Advanced Nerd Notes‚Äã

  • Split VLANs: For homes with smart TVs and IoT devices, separate them into guest VLANs.
  • Device MAC Tracking: Some routers let you assign policies per MAC address (for kids vs adults).
  • Dynamic DNS: If your IP changes often, configure DDNS and bind it to OpenDNS/NextDNS updates.

Bridging the Gap: CAP Theorem for Senior React Developers

May 5, 2025 ¬∑ 5 min read
D Balaji
Lead Design Technologist

Why this post? As frontend engineers, we often focus narrowly on frameworks and tooling‚Äîprimarily JavaScript, React, and UI libraries. But many of us hit a career plateau because we lack exposure to core software engineering principles.

This post is part of a growing genre I call ‚ÄúBridge Posts‚Äù‚Äîconnecting frontend development to foundational software architecture concepts. The goal is to help frontend engineers think like system designers, not just component builders.

Today, we explore the CAP Theorem, a classic principle in distributed systems, and map it to familiar frontend scenarios‚Äîsuch as Git workflows, real-time collaborative UIs, and offline-friendly apps.

Understanding the CAP Theorem‚Äã

In distributed systems, the CAP Theorem states that during a network partition (i.e., some parts of the system can‚Äôt communicate), you can only guarantee two of the following three:

  • Consistency (C): All nodes see the same data at the same time.
  • Availability (A): Every request receives a response‚Äîregardless of the freshness of the data.
  • Partition Tolerance (P): The system continues to operate even when parts of it can‚Äôt communicate.

In practice, partition tolerance is non-negotiable in any distributed system. Therefore, systems must choose between consistency and availability when partitions occur.

Git Analogy: You Already Use CAP‚Äã

Let‚Äôs start with Git‚Äîa tool every developer knows.

  • When you commit locally on a plane, you‚Äôre in a partitioned state.
  • You can continue working (Availability), even though your code may diverge from your teammate‚Äôs (Consistency is compromised).
  • Once reconnected, you merge changes to restore consistency.

Git is an AP (Available + Partition-Tolerant) system. It tolerates partitions and lets you work offline but eventually requires reconciliation.

React Use Case: Real-Time Collaborative Forms‚Äã

Now imagine you‚Äôre building a collaborative form in React. Multiple users edit the same form in real-time. Updates are synchronized via WebSockets or polling.

How does CAP play out here?

1. CP ‚Äì Consistent & Partition-Tolerant‚Äã

  • If a user loses network connectivity, editing is disabled.
  • This ensures everyone always sees the latest state.
  • However, the application becomes unavailable for offline or disconnected users.

Use cases: Healthcare apps, finance platforms‚Äîwhere data integrity is paramount.

2. AP ‚Äì Available & Partition-Tolerant‚Äã

  • Users can continue editing offline.
  • Changes are stored locally and synced later.
  • This may lead to conflicting edits, requiring merge strategies.

Use cases: Note-taking apps, chat applications‚Äîwhere user flow matters more than perfect sync.

3. CA ‚Äì Consistent & Available (No Partition Tolerance)‚Äã

  • Works as expected under perfect network conditions.
  • Any partition causes the system to fail or block.
  • While theoretically ideal, this model is impractical in real-world distributed systems.

üåê Designing for Partition Tolerance‚Äã

A network partition occurs when different components of a distributed system‚Äîclients, services, or databases‚Äîcannot communicate due to a temporary network failure. Each component may still be operational, but they're isolated like islands without bridges.

In frontend development, this is surprisingly common:

  • A user loses internet connectivity mid-session.
  • A mobile app hits a dead spot with no signal.
  • The frontend can reach a CDN or cache but not the main API server.

Designing for Partition Tolerance means your app should continue functioning as gracefully as possible, even during such disconnects.

As a React developer, this involves:‚Äã

  • Storing user actions locally (memory, localStorage, IndexedDB).
  • Queuing mutations and syncing later (e.g., Service Workers, Apollo cache, Redux middleware).
  • Providing clear UI cues: ‚ÄúYou‚Äôre offline, changes will sync later.‚Äù
  • Implementing conflict resolution logic, if needed.

Real-world examples:

  • Figma continues rendering and recording user edits during disconnects.
  • Notion lets you type offline and syncs the block tree later.
  • Gmail stores draft emails offline and sends them once reconnected.

These applications opt for Partition Tolerance, ensuring the app remains usable‚Äîeven if consistency is delayed or temporarily broken.

Designing for Partition Tolerance doesn‚Äôt mean ignoring consistency‚Äîit means accepting that consistency might be eventual, not immediate.

In distributed systems, network failures are not rare edge cases‚Äîthey're expected events. As frontend engineers, acknowledging and designing for them elevates your thinking from component trees to system-level resilience.

Mapping CAP to Frontend Patterns‚Äã

Optimistic UI (e.g., message send)APAssumes success and syncs with the server later.

What This Means for Frontend Developers‚Äã

You don‚Äôt have to be building a distributed database to care about CAP. If your application:

  • Caches remote data
  • Lets users work offline
  • Supports multi-user collaboration
  • Relies on eventual consistency

‚Ä¶then you‚Äôre actively navigating CAP trade-offs.

Ask yourself:

  • Can users work with stale data? ‚Üí Choose Availability.
  • Must every write be accurate and conflict-free? ‚Üí Prioritize Consistency.
  • Should the app always respond‚Äîeven during outages? ‚Üí Design for Partition Tolerance.

Closing Thoughts: Frontend as a Distributed System‚Äã

‚ÄúCAP isn‚Äôt just a backend concern‚Äîit manifests in every interactive, networked UI you build.‚Äù

Whether you‚Äôre building a rich client with React Query, crafting optimistic updates, or designing for offline-first usage, you‚Äôre constantly making trade-offs. Understanding CAP helps you make them consciously.

This post was part of a broader mission to elevate frontend engineers into system thinkers‚Äîdevelopers who don‚Äôt just build buttons, but design resilient user experiences.

Let‚Äôs not be cookie-cutter React developers. Let‚Äôs bridge the gap.

Understanding How LLMs Work with a Doctor's Clinic Analogy

May 1, 2025 ¬∑ 5 min read
D Balaji
Lead Design Technologist

Large Language Models (LLMs) are revolutionizing the way we build intelligent applications‚Äîespecially in frontend development. From ChatGPT to custom AI copilots, LLMs are everywhere. But have you ever wondered how LLMs actually work under the hood?

The theory behind LLMs can be complex‚Äîterms like embeddings, transformers, tokenization, and vector space often feel overwhelming. That‚Äôs why I‚Äôm using a familiar analogy: a visit to a local doctor‚Äôs clinic.

This blog post breaks down the internal architecture of LLMs using a real-world story that‚Äôs easy to visualize and remember. Whether you're a developer exploring AI integration, an engineer curious about embeddings, or someone building chat interfaces, this analogy-driven explanation will help you understand:

  • What tokenization really means
  • Why vectors and embeddings are essential
  • How transformers "think" using attention
  • Why temperature affects LLM creativity
  • How memory and context shape outputs

Let‚Äôs walk into the clinic. üö∂‚Äç‚ôÇÔ∏èüíä

Characters in the Community Clinic Example‚Äã

  • User ‚Äì A person interacting with the LLM (the patient)
  • LLM ‚Äì The large language model (the doctor)
  • Internals ‚Äì Sequence of steps the model uses to generate a response (diagnostic workflow)

Step 1: Patient Describes the Issue to Doctor‚Äã

Scene:
The patient walks into the clinic.

Conversation:

  • Doctor: Hi, please sit down.
  • Patient: Thank you.
  • Doctor: Tell me.
  • Patient: I got back from my friend‚Äôs wedding 2 days back. Since yesterday, I feel cold and have body pains.
  • Doctor: (listening) hmm.

LLM Analogy:
This step is like tokenization ‚Äî breaking the input sentence into smaller units (tokens) that the machine can understand.

Step 1 - Tokenizer:
Converts input from human language to machine-readable tokens.

  • Input: Plain text / story
  • Output: Word tokens (e.g., "wedding", "cold", "pains")

Step 2: Doctor Comprehends the Patient Problem‚Äã

The doctor interprets the patient‚Äôs words and thinks in terms of temperature, symptoms, etc. They may even order diagnostic tests for more data.

LLM Analogy:
This is like the embedding layer, where word tokens are turned into number arrays (vectors) that hold semantic meaning.

Step 2 - Embedding Layer:
Converts tokens into vector form for semantic understanding.

  • Input: Word tokens
  • Output: Vectors that represent meaning (like symptoms turned into medical data)

Step 3: Doctor Runs Diagnosis Internally‚Äã

The doctor thinks through the problem using experience and logic‚Äîchecking symptoms against patterns they‚Äôve seen before.

LLM Analogy:
This is the transformer architecture‚Äîespecially self-attention layers, which compare words across the sentence to extract meaning and decide what matters most.

Step 3 - Transformer Layers:
Deep learning steps like self-attention and feedforward networks.

  • Input: Vectors
  • Output: Context-aware vectors based on internal learned patterns

Step 4: Doctor Considers Constraints (like budget)‚Äã

Doctors don‚Äôt always prescribe expensive tests‚Äîpatient affordability, practicality, and history all affect decisions.

LLM Analogy:
This step is like positional encoding‚Äîensuring the model understands the order and structure of the sentence.

Step 4 - Positional Encoding:
Adds position-related meaning to vectors.

  • Input: Related vector tokens
  • Output: Ordered vector tokens based on sentence position

Step 5: Doctor Writes a Prescription‚Äã

The doctor now documents the diagnosis and treatment. It‚Äôs a direct result of structured thinking and patient input.

LLM Analogy:
This is where the model decodes the internal vector into a predicted sequence of tokens.

Step 5 - Decoder:
Generates output tokens based on model‚Äôs confidence and logic.

  • Input: Positional vectors
  • Output: Probable tokens in correct context

Step 6: Doctor Shares the Prescription with the Patient‚Äã

The final interaction. If the patient is nervous, doctor might tweak the recommendation. This is where human nuance enters.

LLM Analogy:
The LLM now converts tokens back into human-readable words. Here, temperature plays a role in how creative or safe the response is.

Step 6 - Output Layer:
Converts final vector into output text based on temperature setting.

  • Input: Final vector
  • Output: Natural language sentence

Bonus Step: The Doctor Remembers Your Previous Visit‚Äã

LLMs with memory or context windows can remember prior conversations, like a doctor recognizing returning patients. This helps give better, contextual responses in multi-turn dialogues.
